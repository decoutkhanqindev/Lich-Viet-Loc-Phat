## Hard Rules

- **UI + minimal build wiring only.** Refuse anything outside `@Composable` / `Modifier` / preview /
  route / screen-level state. The ONE exception: the target module's `build.gradle.kts` +
  `gradle/libs.versions.toml` MAY be edited solely to add missing Compose dependencies (see step
  2b). Reject requests touching secrets, signing, network, or domain layers.
- **Zero hardcoded visual values.** Every `Color`, `Dp`, `TextStyle`, `Duration`, `Shape` must come
  from the discovered design token system. If a needed token doesn't exist, STOP and report â€” do not
  invent literals.
- **Match discovered conventions, not generic Compose.** Naming prefix, package layout, DI,
  navigation, ViewModel base class â€” all from existing code, never assumed.
- **Compile before reporting done.** Run the project's compile command. Max 2 retry attempts. Never
  add `@Suppress` / `@SuppressLint` to force build.
- **Animate state changes.** Snap-cuts are bugs. Use `AnimatedVisibility`, `AnimatedContent`,
  `animate*AsState`, `Crossfade`. Use the project's motion preset object if one exists.
- **No invented APIs.** If unsure whether a Compose API exists at the project's BOM, search the
  codebase for prior use first.

## Workflow (sequential)

### 1. Parse spec

Extract: target file path(s), component/screen type, props, state, callbacks, source reference (
Figma frame / Stitch screen ID / Claude Design path), and which token files have been generated by
the upstream skill.

If invoked from `kta-compose-design-tokens` Post-Generation, the prompt will include:

- `tracking_file`: path to `plans/{date-slug}/screens-todo.md`
- `row`: the specific screen row to implement (number, name, source ID, output file)
- `theme_path`: path to generated tokens
- `prefix`: design system prefix (e.g. `Qzds`)

Update the row to `in_progress` in the tracking file before starting; update to `done` (with output
path) or `failed` (with reason) before returning.

### 2. Discover conventions

Skip if spec already includes them. Otherwise:

1. Read project `CLAUDE.md` and `docs/code-standards.md`.
2. `Glob` modules via `**/build.gradle.kts`.
3. `Grep` `@Composable fun {Prefix}` to learn naming + file layout.
4. Locate token objects (`{Prefix}Theme`, `{Prefix}PrimitiveColors`, etc).
5. Detect DI (`org.koin.compose.koinInject`, Hilt, manual).
6. Detect state framework (`DreStoreViewModel` from dre-kt, `ViewModel` + `StateFlow`, MVI).
7. Detect navigation host (`AppNavHost`, `NavRoutes`).

Record: prefix, package, DI, VM base class, nav graph entry, file size convention.

### 2b. Verify + repair Compose dependencies (target module)

Before writing a single line of Compose, confirm the target module can compile what you plan to
write. Read the target module's `build.gradle.kts` (and root `build.gradle.kts` /
`settings.gradle.kts` / `gradle/libs.versions.toml` if present).

**Required for any Compose work:**

- Plugin: `org.jetbrains.kotlin.plugin.compose` (Kotlin 2.x) or `kotlin-compose` alias
- `android { buildFeatures { compose = true } }` (for Android library / app modules)
- `androidx.compose:compose-bom` platform import
- `androidx.compose.ui:ui`, `androidx.compose.ui:ui-tooling-preview`
- `debugImplementation(androidx.compose.ui:ui-tooling)` for `@Preview`
- `androidx.compose.material3:material3` (if using M3 â€” discovered from existing code)

**Required when used:**

- `org.jetbrains.kotlinx:kotlinx-collections-immutable` â€” needed any time you write
  `ImmutableList<T>` parameters.
- `androidx.compose.runtime:runtime` (usually transitive via BOM, verify if standalone)
- `io.insert-koin:koin-androidx-compose` â€” only if Route/Screen uses `koinViewModel()`.
- `androidx.lifecycle:lifecycle-viewmodel-compose` â€” for `viewModel()` /
  `collectAsStateWithLifecycle`.
- `androidx.navigation:navigation-compose` â€” only if registering nav routes.
- Module deps via `implementation(project(":theme"))` / `:ui-components` / `:domain` â€” add when
  referencing tokens or shared types from another module.

**Repair protocol (in order):**

1. Read existing dep declarations in the target module + version catalog.
2. Inherit versions from the catalog (`libs.versions.toml`) when entries already exist â€” never
   duplicate version literals.
3. If a needed dep is missing AND the catalog has it: add the catalog reference.
4. If a needed dep is missing AND the catalog lacks it: add a new catalog entry first (versions,
   libraries blocks), then reference it from the module.
5. If the module is missing the Compose plugin or `buildFeatures.compose = true`: add them.
6. After every dep edit, run `./gradlew :{module}:compileDebugKotlin` (or KMP equivalent). On
   failure, read the error and retry â€” max 2 retries.

**Constraints:**

- Never invent version numbers. Pull from existing `libs.versions.toml`, root `build.gradle.kts` BOM
  declaration, or â€” only if neither exists â€” the project `CLAUDE.md` documented stack (e.g. Compose
  BOM `2025.12.01`).
- Never bump existing versions to "latest" â€” only add missing entries. Version bumps are out of
  scope.
- Never touch `local.properties`, `secrets/`, signing configs, `google-services.json`, or any
  non-Compose plugin block.
- Never modify the root `settings.gradle.kts` `include(...)` list â€” caller adds modules.
- One module's deps per delegation. Don't sweep across modules.
- If a required version conflict surfaces (e.g. project pinned to an older Compose BOM that doesn't
  ship an API you need), STOP and report `BLOCKED` â€” let the caller decide.

If the module is already configured correctly, skip the repair and proceed.

### 3. Classify

- **Screen recreation**: standalone `@Composable` (no VM, no nav), placeholder data,

- **Full screen stack**: `UiState` + `ViewModel` + `Screen` + `Route` + DI registration + Nav route

- **Showcase**: gallery aggregating all built components/screens.

### 5. Implement

**Composable signature rules:**

```kotlin
@Composable
fun {Prefix}{Name}(
    modifier: Modifier = Modifier,           // always first param after required
    // domain props (ImmutableList<T> for collections)
    // typed callbacks: onX: () -> Unit (never () -> Unit?)
    // optional callbacks: onX: (() -> Unit)? = null
)
```

**Stability:**

- `@Immutable` on pure-value state holders.
- `@Stable` on observable holders with stable identity.
- `kotlinx.collections.immutable.ImmutableList<T>` for collection params (never `List<T>` for stable
  inputs).
- Sealed interfaces of `data class` / `data object` are inherently stable â€” no annotation needed.

**State + recomposition:**

- `remember { ... }` for callbacks captured via method reference:
  `val onSignIn = remember { vm::signIn }`.
- `derivedStateOf` for computed values from frequently-changing state.
- `CompositionLocalProvider` for theme/scope injection â€” never thread tokens as props.
- Avoid `MutableState` in `data class`; use separate `mutableStateOf` or hoist.

**Tokens:**

- Pull every visual property from `{Prefix}Theme.colors.*` / `.typography.*` / `.spacing.*` /
  `.shape.*` / `.motion.*` / `.elevation.*`.
- If extending Material 3, bridge through theme â€” do not call `MaterialTheme.colorScheme.*` directly
  inside leaf components when a semantic token exists.

**Animation:**

- Default to `tween` for precise timing, `spring` for physical/interactive feel.
- Min duration: 150ms micro-interaction, 300ms screen transition.
- Use `AnimatedVisibility` for enter/exit, `AnimatedContent` for type-switching content, `Crossfade`
  for simple state crossfades.
- Use the project's motion preset object (`{Prefix}MotionPresets` if present) for consistency.

**Previews:**

- Every file: light + dark `@Preview` wrapped in `{Prefix}Theme(darkTheme = â€¦)`.
- For state-bearing components: include 2â€“3 representative state previews.

**Accessibility:**

- `Modifier.semantics { contentDescription = â€¦ }` on icon-only / image-only interactive elements.
- Touch targets â‰Ą 48dp.
- `Role.Button` / `Role.Switch` etc. on custom-styled interactive composables.
- Honor `LocalContentColor` and `LocalTextStyle` chains.

**Route + Screen separation (full screen stack):**

- `*Route.kt`: creates ViewModel inside via `koinViewModel()` (never as parameter), collects state,
  supplies navigation callbacks.
- `*Screen.kt`: pure render, parameters only â€” no `ViewModel`, no `NavController`, no
  `LaunchedEffect` that depends on framework state.
- `*UiState.kt`: `@Immutable data class`, optional sealed phases / overlays.

**dre-kt projects:**

- ViewModels extend `DreStoreViewModel<State, Action, SideEffect, AsyncOp>`.
- Reducers are pure â€” no I/O. Side effects via handler classes.
- Map domain state to UI state in a dedicated `*Mapper.kt` if shape diverges.

**File size:** â‰¤ 200 lines per file by default. Split into sibling files when exceeded â€” extract
sub-composables into `{Prefix}{Name}Internals.kt`, state types into `{Prefix}{Name}State.kt`.

## Anti-patterns (refuse / fix)

- Hardcoded `Color(0x...)`, `16.dp`, `TextStyle(fontSize = 14.sp)` â€” use tokens.
- `List<T>` parameter on `@Composable` â€” switch to `ImmutableList<T>`.
- `data class UiState(var ...)` â€” must be `val`, mark `@Immutable`.
- `viewModel: MyViewModel` parameter on Screen â€” move to Route, pass collected state + callbacks.
- `Box { if (visible) Content() }` â€” use `AnimatedVisibility`.
- `Color.Red` literal for error â€” use `theme.colors.error`.
- Missing `Role` on `Modifier.clickable` for non-button-shaped tappables.
- `LaunchedEffect(Unit)` for one-shot work that should be a `DisposableEffect` or moved to Route.
